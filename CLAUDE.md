## Architecture overview

SAPIENT is a pure-Rust edge LLM inference engine. The crates form a clear dependency stack:

```
sapient-cli              ← the `sapient` binary (chat, transcribe, speak, converse, stats, pull, run, models, update …)
  sapient-generate       ← Pipeline API (from_pretrained / from_gguf, generate, chat, stream) + TranscribePipeline + SpeakPipeline
    sapient-models       ← forward engines (LLM) + AudioEngine (Whisper) + SnacDecoder (TTS) + weight loading
    sapient-hub          ← HF Hub client, curated registry, ModelInfo, WhisperConfig
    sapient-tokenizers   ← tokenizer + chat templates + WhisperTokenizer
    sapient-audio        ← audio front-end: decode/resample (symphonia+rubato) + log-mel STFT (realfft)
    sapient-io           ← file format loaders (safetensors mmap, GGUF quant, ONNX)
    sapient-backends-cpu ← CPU kernels (matmul, attention, conv2d, RoPE, quant dot-products)
    sapient-backends-metal ← MLX/Metal GPU backend (optional, `--features mlx`)
    sapient-backends-wgpu ← cross-platform GPU backend via wgpu (Vulkan/DX12/Metal) — Intel/AMD/Nvidia/Apple
  sapient-core           ← Tensor, DType, Shape, Buffer — used by everyone
```

`sapient-backends-wgpu` is the portable GPU path for non-Apple hardware (Intel/AMD/Nvidia on Linux/Windows). It is now **wired into `ForwardEngine`** as `WgpuForwardEngine` (built behind the `wgpu` feature, selected via `--backend wgpu`). The crate provides `WgpuContext` (device acquisition with adapter-max limits + `SHADER_F16` + a pipeline cache) and a GPU-resident compute layer (`resident.rs` + `quant.rs` + `shaders/*.wgsl`): `GpuBuffer` (f32), `GpuQ8Buffer`/`GpuQ4KBuffer`/`GpuQ6KBuffer` (Q8_0/Q4_K/Q6_K kept quantized on-device — a Q4_K_M GGUF loads fully quantized), and the kernels `rms_norm`, `matmul_nt` (GEMV), `matmul_nt_q8_0`/`matmul_nt_q4_k`/`matmul_nt_q6_k` (in-shader dequant GEMV), `rope`, `attention` (causal GQA FlashDecoding), `swiglu`/`add`, `embed` (+`_q8_0`/`_q4_k`/`_q6_k`), plus `copy_range` (KV-cache append). Each kernel is validated bit-close to a CPU reference (`tests/resident.rs`). Develop/test the WGSL on any GPU including Apple Silicon (wgpu uses Metal there); the same shaders run on Intel/AMD via Vulkan. See `docs/ROADMAP.md` Phase 3b for the remaining P5 work (f16/quant KV cache, kernel fusion, batched prefill).

`sapient-ir`, `sapient-runtime`, `sapient-scheduler`, `sapient-telemetry` power a separate *graph-execution* path used internally. The chat/generate path does **not** go through the IR; it uses the native forward engines directly. `sapient serve` (OpenAI-compatible HTTP server) is implemented in `sapient-cli/src/server.rs` and drives the `Pipeline` API directly — it does not use the IR runtime.

## Key design decisions to know

### Four forward engines wired to `Pipeline`
- **`LlamaForward`** — Llama, Mistral, Qwen2.5 (adds q/k/v biases), SmolLM2, TinyLlama (CPU + hybrid Metal), **and Mixtral-class sparse MoE** (per-layer `Ffn::{Dense, Moe}` branch — see the MoE section below)
- **`PhiForward`** — Phi-1/1.5/2 (LayerNorm+bias, parallel block, `partial_rotary_factor`) and Phi-3 (SwiGLU sequential path)
- **`MlxForwardEngine`** (`forward/mlx_engine.rs`, `cfg(macos + feature="mlx")`) — native lazy-graph Metal path for Llama/Qwen GGUF models. Auto-selected in `ForwardEngine::from_gguf_weights` when `use_mlx_engine(backend)` is true (Metal/Auto on Apple Silicon). All activations stay as `mlx_rs::Array`; one `mlx_rs::transforms::eval()` per decode step materialises logits + the whole KV cache. v0.3.5: ~187 tok/s decode + 21 ms TTFT on Qwen2.5-0.5B Q4 (9.4× the CPU path; lowest TTFT of any engine measured).
- **`WgpuForwardEngine`** (`forward/wgpu_engine.rs`, `cfg(feature="wgpu")`) — cross-platform GPU path (Vulkan/DX12/Metal via wgpu) for Llama/Qwen/Mistral, selected via `--backend wgpu`. GPU-resident weights + KV cache; decode runs fully on-device, only logits read back. The portable answer for Intel/AMD/Nvidia on Linux/Windows where MLX (Apple-only) can't run. See its invariants below.

Architecture builder files in `sapient-models/src/architectures/` (gemma, gpt2, bert, mixtral …) build IR graphs for the graph-execution path; they are **not** used for live inference. (The `architectures/mixtral.rs` IR builder is legacy/unwired — live Mixtral runs through `LlamaForward`'s MoE branch below, not this.)

### Mixtral-class sparse Mixture-of-Experts (MoE)
Live MoE is a per-layer FFN branch **inside `LlamaForward`**, not a separate engine — attention, KV cache, RoPE, and hybrid split are all shared with the dense path; only the FFN sub-block differs (`forward/llama.rs`).
- **Detection is by config, NOT `ArchType`.** A Mixtral GGUF reports `general.architecture = "llama"` (llama.cpp folds Mixtral into the llama arch — so it also correctly gets the `unpermute_llama_gguf_qk` q/k un-permute), and a Qwen-MoE safetensors repo is `ArchType::Qwen`. MoE is detected by `info.is_moe()` (i.e. `ModelInfo.moe: Option<MoeConfig>` is `Some`), populated from config.json (`num_local_experts`/`n_routed_experts`, `num_experts_per_tok`, `moe_intermediate_size`, `n_shared_experts`, `first_k_dense_replace`, `norm_topk_prob`, `scoring_func`) **and** GGUF metadata (`{arch}.expert_count`/`.expert_used_count`/`.expert_feed_forward_length`/`.expert_shared_count`/`.leading_dense_block_count`/`.expert_weights_norm`/`.expert_gating_func`). `raw` is `#[serde(skip)]` and Null for GGUF, so MoE config MUST be first-class `ModelInfo` fields — never read it from `raw`.
- **Routing (Mixtral, `route_topk`):** softmax over **all** experts → top-k by softmax value → **renormalise** the k weights to sum to 1 (`norm_topk_prob`). This exact order matters — reordering degrades quality *silently* (coherent-but-wrong tokens), so it's gated by a hand-computed numeric unit test, not just "does it emit words". `is_moe_layer(idx) = moe && idx >= first_k_dense` (Mixtral `first_k_dense = 0` → all layers MoE; DeepSeek/GLM have leading dense layers).
- **Expert compute (`moe_ffn`):** expert-grouped batching — the router assigns tokens to experts, then each **active** expert runs its SwiGLU (`w1`=gate_proj, `w3`=up_proj, `w2`=down_proj) once over its tokens (`[1, rows, hidden]` via `bk.linear_3d`), weighted-scattered back. For decode (`seq==1`) this touches exactly `top_k` experts — the whole point of MoE (top_k-expert bandwidth, not all-expert).
- **Backend gating:** MoE runs **CPU-only** for now. `build_moe_cpu` (`forward/mod.rs`) forces the CPU `LlamaForward`; explicit `--backend metal`/`wgpu` **errors clearly**, `Auto` on an accelerated build silently falls back to CPU with a warning (MLX/`MlxForwardEngine` and wgpu know nothing about experts — routing a Mixtral there would crash/garbage).
- **Two GGUF expert formats (both verified against real files):** (1) **NEWER stacked** — `blk.L.ffn_{gate,up,down}_exps.weight` is all experts in one 3-D blob `[d0, d1, n_expert]` with the expert axis **outermost** (Qwen3-30B-A3B: `ffn_gate_exps` `[2048, 768, 128]`); `split_moe_gguf_experts` (`gguf_weights.rs`) byte-slices it per-expert (the split is dtype-uniform `byte_count`-based like `slice_rows`, since `slice_axis`/`element_size` are invalid for quant dtypes — the Q6_K/embedding-gather bug class), each expert reshaped to `[d1, d0]` = HF `[out, in]`, valid because `d0·d1` is a whole number of quant blocks and the expert axis is contiguous. (2) **OLDER per-expert** (TheBloke Mixtral) — `blk.L.ffn_{gate,up,down}.{e}.weight` are already per-expert 2-D tensors, mapped straight to the canonical key (no split). Both normalise to `layers.{i}.block_sparse_moe.{gate, experts.{e}.{w1,w3,w2}}` so the forward path is format-agnostic and safetensors falls out free. **The stacked split is ZERO-COPY:** each per-expert tensor is a `Tensor::from_buffer` view sharing the stacked buffer at byte offset `e·eb` (`t.offset() + e·eb`) — no expert bytes are copied, so an mmap'd MoE stays mmap-backed (a 63 GB GLM-4.5-Air measured 119 GB peak RSS + 0.34 tok/s with a byte-copy → 118 GB + **2.45 tok/s** as zero-copy views: the copy pinned ~57 GB of experts in non-evictable heap; the views are evictable page-cache). Valid because quant `as_bytes` bounds the view by `byte_count(numel)` (F32/F16 views are UNbounded — a real trap, but GGUF experts are always quantized). The views stay `is_mmap()` so the aarch64 R4 repack skips them (fine — R4 doesn't help m=1 decode).
- **MoE models default to mmap** (`pipeline.rs`: `use_mmap |= model_info.is_moe()`). The heap GGUF loader copies each tensor out of a whole-file buffer (~2× file peak — **49 GB for the 26 GB Mixtral**), which OOMs edge devices; mmap makes weights zero-copy (**RSS ≈ file size, 25.6 GB**) and, as a bonus, skips the Q4_K→R4 expert repack — which measured *slower* for m=1 MoE decode (repack helps prefill, not decode). Force off with `--mmap`-family opts only if needed.
- **Perf (Mixtral-8x7B Q4_K_M, Jetson AGX Thor 14×Neoverse, vs llama.cpp same file same threads, 2026-07-05):** decode **5.5 tok/s** vs llama.cpp 9.95 (1.8×); prefill ~6–9 vs ~15; RSS 25.6 GB (mmap) ≈ file size; **0 quality loss** (temp-0 greedy token-identical to llama.cpp for ~28 tokens, then f32-order drift). **The gap is NOT MoE** — a dense control (Qwen2.5-1.5B, same Q4_K+Q6_K profile) is 3.16× (bigger), decomposing to **~1.94× single-core kernel quality (llama.cpp = KleidiAI microkernels) × ~1.6× multicore scaling** (per-GEMV rayon fork/join). **SVE is a dead end on Thor** (128-bit = NEON width; nightly `stdarch_aarch64_sve` works but pointless); task-count is already optimal (`SAPIENT_GEMV_TPC` sweep). Closing parity = KleidiAI-class NEON kernels + a lower-overhead decode threadpool (deep work, see ROADMAP + `docs/BENCHMARKS.md`). Bench: `sapient-generate/tests/moe_bench.rs` (ignored). llama.cpp can't load per-expert Mixtral GGUFs anymore (needs old `b1928`); SAPIENT loads both layouts.
- **Router precision:** the router gate (`block_sparse_moe.gate`) is skipped by `should_quantize_online` (kept F16/F32 — routing flips under Q8_0 rounding, matching llama.cpp). Experts ARE online-quantized (safetensors F16→Q8_0) and, on aarch64, repacked to `Q4_K_R4` and run through the multi-row SDOT/SMMLA kernels via `moe_ffn`.
- **Extension points (parsed, but the engine bails loudly at load — `validate_moe_support`):** sigmoid gating (DeepSeek/GLM `noaux_tc` group-limited routing) and shared experts (`n_shared_experts`). GLM-4.5-Air is the next tractable target (partial-RoPE GQA + softmax MoE, fits Thor at Q4); **GLM-5.2 is out of scope** (~750B params/1.5 TB, and needs MLA + DeepSeek Sparse Attention — three engine projects on top of MoE).
- **Registry:** `openhorizon/mixtral-8x7b-q4` → `TheBloke/Mixtral-8x7B-Instruct-v0.1-GGUF` (per-expert format, Q4_K_M ≈ 26 GB, needs 32 GB+ RAM). Tokenizer fallback: `mixtral` → `mistralai/Mistral-7B-Instruct-v0.2` (32000 vocab + `[INST]` template — "mixtral" does NOT contain "mistral", so it needs its own arm and must NOT fall through to the v0.3/32768 default → vocab-compat failure).
- **Tests:** `route_topk` numeric unit tests (`forward/llama.rs`); `tests/moe_coherence.rs` — identical-experts≡dense (real attention), distinct-experts vs a hand-written reference (attention neutralised via `v_proj=0`), and Q4_K identical-experts (exercises the aarch64 R4 repack, transitively valid via `cpu_repack.rs`); GGUF name-map + `split_moe_gguf_experts` byte-slice tests (`gguf_weights.rs`); `sapient-generate/tests/moe_e2e.rs` (**ignored** — downloads Mixtral, greedy → "Paris").
- **Verification status (be honest):** the mechanism is synthetically gated + the GGUF formats are **dim-verified against real headers** (TheBloke Mixtral per-expert `ffn_gate.{e}` `[4096,14336]`; Qwen3-30B-A3B stacked `ffn_gate_exps` `[2048,768,128]`), but the **golden real-decode gate (`moe_e2e`) has NOT been run** — it needs a 32 GB+ box (`cargo test -p sapient-generate --test moe_e2e --release -- --ignored`). Both the per-expert (Mixtral) and stacked (GLM-4.5-Air) paths are now **decode-verified end-to-end on Thor** (coherent output — see below).

### GLM-4.5-Air (`Glm4Moe`) — the sigmoid-gate MoE, on top of the Mixtral foundation
`Glm4MoeForCausalLM` / GGUF arch `glm4moe` (106B-A12B). Extends `LlamaForward`'s MoE branch — **not** a new engine. **Verified end-to-end on a Jetson AGX Thor** (pure Rust, zero CUDA): coherent output, decode **2.45 tok/s**, prompt→"The Roman Empire, one of history's most influential civilizations…". The deltas from Mixtral, all in the shared MoE/attention path:
- **Sigmoid gate + correction bias + shared expert** (`route_topk` `MoeScoring::Sigmoid`): `sigmoid(logits)` are the WEIGHTS; **selection** is `sigmoid + e_score_correction_bias` (the DeepSeek-V3 aux-loss-free bias steers *which* experts win, NOT their weights); renorm (`+1e-20`), `× routed_scaling_factor`. Plus an always-on **shared expert** (unweighted, run over every token). `n_group > 1` (group-limited routing, DeepSeek-V3/GLM-5.2) still bails.
- **Partial RoPE 0.5** — `forward_layer` uses `apply_rope_partial` when `partial_rotary_factor < 1.0` (64 of 128 head dims rotate). Dense/Qwen (1.0) unchanged.
- **`ArchType::Glm4Moe`** — distinct from `Llama` so the q/k unpermute correctly **skips** it (GLM is NEOX). Sigmoid inferred from `model_type`/`expert_gating_func=2` (config omits `scoring_func`).
- **GGUF specifics (verified from a real header):** `head_dim` from `attention.key_length` (128, ≠ hidden/heads = 42 — getting this wrong panics the q reshape); **MTP-layer cap** `num_hidden_layers = block_count − nextn_predict_layers` (47−1=46, skip the multi-token-prediction head); q/k/v biases but no o-bias; `post_attention_norm`→FFN norm, `exp_probs_b`→correction bias, `*_shexp`→shared expert; stacked experts reuse the zero-copy split.
- **Split-GGUF** — GLM-4.5-Air Q4_K_M is 2 shards (~63 GB, `split.count=2`, > HF's 50 GB limit). `gguf_split_shards` (sapient-hub) parses `-NNNNN-of-MMMMM.gguf`; the client downloads all shards, `load_gguf_hf_weights` auto-discovers siblings and merges. **Pipeline routing gate is "all paths `.gguf`", NOT `len==1`** (the split-GGUF bug that first surfaced on Thor).
- **Registry** `openhorizon/glm-4.5-air-q4` → `unsloth/GLM-4.5-Air-GGUF` (needs a 96 GB+ device); tokenizer fallback `glm-4.5`/`glm4moe`→`zai-org/GLM-4.5-Air` (ungated, tokenizer-only download — vocab 151552).
- **Tests:** `route_topk_glm_sigmoid_gate` (numeric, bias-steers-selection-not-weights) + `glm_moe_matches_reference` (full-forward hand-oracle) + `partial_rope_changes_logits` + `parse_glm4_moe_config` + `maps_glm4moe_gguf_names` + `gguf_split_shards_*` + `q8_0_quantize_roundtrips`. Five real-model bugs this caught that synthetic tests couldn't: split-routing gate, GGUF head_dim, MTP cap, heap-copy RSS, and the Q5_0-F32-expansion RSS below.
- **Q5_0→Q8_0 at load (RSS fix):** unsloth "dynamic" quants store some tensors as **Q5_0** (24 `ffn_down_exps` in GLM-4.5-Air), a type SAPIENT can't keep as blocks (`to_sapient_dtype` returns None). The loader (`make_tensor`/`make_tensor_mmap`, sapient-io) previously F32-expanded them (738M-param tensor → 2.95 GB each = **70 GB heap**). Now any quantized-but-unsupported type is **re-quantized to Q8_0** at load (`quantize_to_q8_0`, near-lossless since 8⊇5 bits, ~1.06 B/weight). **Peak RSS 118 → 72 GB, heap 66 → 18 GB, decode 2.45 → 3.23 tok/s (+32%), prefill 0.80 → 3.94 (5×)** — the memory-pressure relief also sped decode. Now fits a 96 GB box. Output byte-identical (verified on Thor). Remaining heap (~18 GB) is those Q8_0-converted experts; a first-class Q5_0 mmap dtype+kernel would zero it (future).

### Whisper speech-to-text (audio path, `sapient transcribe`)
On-device STT is a **separate engine path** from the text `ForwardEngine` — it never touches the chat/serve code (zero regression risk). Components:
- **`sapient-audio`** (`mel.rs`, `io.rs`, `config.rs`): CPU-only front-end. `io::load_audio` decodes any `symphonia` format → mono `f32` and resamples to 16 kHz via `rubato`. `MelFrontend::log_mel` computes the Whisper log-mel (`realfft` STFT, n_fft=400/hop=160, periodic Hann, **slaney** mel filterbank built analytically to match OpenAI/librosa, then `log10`→clamp to `max-8`→`(x+4)/4`). Output `[1, n_mels, 3000]`. Runs once per 30 s chunk, so it stays CPU even when the body is on GPU.
- **`WhisperForward` + `AudioEngine`** (`forward/whisper.rs`): encoder (`conv1`→`conv2`→ +`embed_positions` → N pre-LN blocks → `layer_norm`) and decoder (`embed_tokens`+`embed_positions` → N blocks of [causal self-attn + cross-attn + MLP] → `layer_norm` → tied `proj_out`). Reuses `LlmBackendDispatch` for linear/layernorm+bias/add; **attention calls the CPU flash kernel directly** because non-causal attention needs an explicit mask. `AudioEngine::encode` runs the encoder once and caches each decoder layer's cross-attn K/V (`set_audio_context`); `decode_step` keeps a growing per-layer self-attn KV cache.
- **Critical invariants:**
  - **Mask, not `None`:** the CPU `scaled_dot_product_attention` treats `mask=None` as **causal**. The encoder self-attn and cross-attn MUST pass an explicit **all-zeros** `[seq_q,seq_k]` mask (`zeros_mask`); decoder self-attn passes `causal_mask(seq, total)`. Using `None` for the encoder silently makes it causal → wrong transcripts.
  - **Exact GELU:** Whisper uses erf GELU, not the tanh approx. Use `kernels::elementwise::gelu_erf` (added for this), not `gelu`.
  - **conv1d** = `conv2d` with a height-1 image (`forward/conv.rs`); conv weights stay full precision.
  - **Online Q8_0** quantization applies only to F16/BF16 2-D linears (`should_quantize_online` skips conv/embed/pos/norm/bias by name+rank). F32 weights pass through unquantized — which is why the synthetic coherence test (F32) is an exact comparison.
- **`WhisperConfig`** (`sapient-hub/src/whisper_config.rs`): dual encoder/decoder dims — `ModelInfo` (single-stack) can't represent Whisper, so `TranscribePipeline` branches on `ArchType::Whisper` and parses this instead.
- **`WhisperTokenizer`** (`sapient-tokenizers/src/whisper.rs`): GPT-2 BPE + control tokens; `sot_sequence(lang, translate, timestamps)` builds the forced prompt; language auto-detect = argmax over `<|lang|>` token ids from one decode step after `<|sot|>`.
- **`TranscribePipeline`** (`sapient-generate/src/transcribe.rs`): HF safetensors load → 30 s chunking → mel → encode → forced-prompt greedy decode (drops control/timestamp tokens, stops on `<|endoftext|>`).
- **Loading:** ship **HF safetensors** (`openai/whisper-{tiny,base,small}`, registry family `Whisper`); whisper.cpp q5_0/q5_1 GGUF don't map to SAPIENT's Q-schemes, so safetensors + online-Q8_0 is the path.
- **Tests:** `sapient-audio` mel/STFT/resample tests; `forward/conv.rs` conv1d-vs-naive; `tests/whisper_coherence.rs` (synthetic tiny model, batch==incremental decode equivalence + cross-attn dependency); `tests/transcribe_e2e.rs` (ignored — downloads `whisper-tiny`, transcribes the JFK clip, asserts "country"). **First-cut scope:** greedy decode, fixed 30 s windows, no timestamps/beam/suppress-tokens (TTS/STS are `docs/ROADMAP.md` Phase 6d+).
- **`WhisperWgpuEngine`** (`forward/whisper_wgpu.rs`, `cfg(feature="wgpu")`, `--backend wgpu`) — the cross-platform GPU path. Mirrors `WhisperForward` on wgpu: weights upload once as **f32** (no online-Q8_0 — the GPU matmul reads f32, so `TranscribePipeline` feeds it the *raw* weight map via `build_audio_engine`), encoder + decoder run on-device, only logits read back. mel/STFT/conv stay CPU. New wgpu kernels it needs (`sapient-backends-wgpu`): `layer_norm` (with bias), `gelu_erf` (elementwise op=2), `add_bias` (broadcast, op=3), `transpose_heads` (matmul layout `[seq,H,hd]` ↔ attention layout `[H,seq,hd]`), and a **`causal` flag on `attention`** (false for non-causal encoder self-attn + cross-attn; true for decoder self-attn). Selected via `AudioEngine::WhisperWgpu`. Coherence-gated vs the CPU engine in `tests/whisper_wgpu_coherence.rs` (encoder + logits + argmax, max_err < 5e-3); verified end-to-end (JFK transcript identical to CPU). Mac→Metal validated locally; CI runs the same WGSL on Vulkan/DX12.
- **STT decode polish (6c)** — all in `transcribe.rs`: `suppress_tokens`/`begin_suppress_tokens` from `generation_config.json` (fetched optionally by the hub client → `WhisperGenConfig`); **streaming** (`transcribe_stream`/`transcribe_samples_stream`, stable-delta emission); **timestamps + long-audio re-seek** (`--timestamps`, `apply_timestamp_rules` = greedy subset of openai-whisper's, advance window by last segment-end); **beam search** (`--beam-size`, prefix-replay via `decode_chunk_beam`, length-normalized). Pure helpers unit-tested. Serve: **`POST /v1/audio/transcriptions`** (`server.rs`, OpenAI multipart) uses a parallel `ModelCache<Arc<TranscribePipeline>>` + `get_or_load_audio`, sharing `inference_sem`.
- **Speech-to-speech (6e)** — `ConversePipeline` (`sapient-generate/src/converse.rs`): STT→LLM(`chat` + history)→`Tts`→audio, device-agnostic (`&[f32]` in / `Vec<f32>` out), so voice-in/text-out works today via `NoopTts` and is WAV-injection-testable. Real-time loop = `sapient converse` (`cfg(feature="audio-io")`): `sapient-audio` `MicCapture`/`SpeakerPlayback` (cpal 0.15, mono f32 over flume, realtime callback drop-on-overrun, `Stream` is !Send so kept on the driving thread) + `EnergyVad` (utterance segmenter) + `SentenceChunker` (LLM→TTS sentence streaming). The Phase-6d Llama→SNAC TTS plugs into the same `Tts` trait (and `SpeakPipeline` now implements `Tts`, wired via `sapient converse --speak` for spoken replies — slow on CPU since it's a 3B LM per reply). **Live CLI UX (`converse_command`):** requests mic permission up front (`sapient_audio::permissions::request_microphone` — real AVFoundation/TCC prompt on macOS; Windows/Linux have no per-app prompt → `MicPermission::Unknown` + `microphone_guidance()`/`open_privacy_settings()`); shows a TTY-gated live **mic level meter** (so a silent/denied mic is visible), a "transcribing…" status, and **streams the LLM reply token-by-token** via `ConversePipeline::{transcribe_utterance, respond_streaming}`; warns once if ~5 s pass with no mic signal; drains the mic during STT/LLM/TTS to avoid self-trigger/echo. **`audio-io` is now a default feature of `sapient-cli`** (v0.4.3) so `converse` ships in the official macOS/Windows/x86_64-Linux binaries out of the box — macOS/Windows need no system libs (CoreAudio/WASAPI); **Linux needs `libasound2-dev` at build time** (CI/release Linux jobs install it). The **aarch64-linux release now ships `converse`** too: it's built **natively on a GitHub-hosted ARM runner (`ubuntu-22.04-arm`)** with default features (cpal/ALSA compile natively there), replacing the old zigbuild cross that had to drop `audio-io`. So a Pi 5 gets `converse` from `sapient update` out of the box. The `#[cfg(not(feature="audio-io"))] converse_command` is the bail-out stub only for explicit `--no-default-features` builds. **Spoken-voice engine is selectable: `converse --speak [--tts kokoro|orpheus]`** (default `kokoro`, real-time; `orpheus` for the richer-but-slow 3B voice). `--stt` is validated to be a Whisper model (a TTS model there now errors clearly instead of "no weights found").
### Backend selection follows the compiled binary (`Auto`)
`LlmBackendKind::Auto` resolves to the binary's compiled accelerator so a GPU build runs on the GPU with no flag: **MLX/Metal** when `use_mlx_engine(Auto)` (macOS + `mlx` feature), else **wgpu** when `use_wgpu_engine(Auto)` (the `wgpu` feature = `-gpu` release variant), else **CPU** (`forward/mod.rs`, both the GGUF and safetensors selection paths; MLX is preferred before wgpu). `chat`/`transcribe`/`speak`/`converse`/`serve` all default to `--backend auto`, so STT+LLM+TTS use the GPU on a GPU binary and CPU on a CPU binary. `converse`'s banner prints the resolved label via `ConversePipeline::backend_label()` → `Pipeline::backend_display_label()` (green when accelerated; yellow + "run `sapient update --metal`" hint on CPU). Whisper STT has no MLX path yet, so on the metal build STT stays CPU (LLM+TTS get Metal); the wgpu build accelerates STT too (`WhisperWgpuEngine`). **Perf reality:** Orpheus is a 3B autoregressive LM — even on Metal it decodes ~17 codec tok/s and SNAC needs ~82 for real-time, so spoken Orpheus replies are *not* real-time. **This is solved by Kokoro-82M** (non-autoregressive, real-time on CPU — see its section), which `converse --speak` now uses by default; Kokoro is CPU-only so it doesn't need the Metal build for real-time. Orpheus remains available via `sapient speak orpheus-3b` for its voice.
### Streaming voice loop (Phase 10 first cut — `feat/streaming-voice`)
Perceived latency (end of user speech → first reply audio) dropped ~4.4 s → ~3.1 s on M4 CPU via three changes, all measured with the new per-turn instrumentation. (1) **`LiveStt`** (`converse.rs`): a background incremental transcriber — the CLI feeds it `EnergyVad::speech_so_far()` snapshots every ~0.5 s *while the user is talking* (new VAD taps `in_speech`/`speech_so_far`; `TranscribePipeline`/`MelFrontend` are now `Clone`, engine shared via its internal `Arc<Mutex>`); at utterance end `settle()` returns the latest transcript + samples covered, and if only the ~400 ms hangover is new audio the transcript is used directly — **STT leaves the critical path** (gated by `tests/live_stt_e2e.rs`: snapshot-fed JFK == full-pass transcript). (2) **Early-first-clause TTS**: `SentenceChunker::with_early_first(24)` lets the FIRST chunk break at a clause boundary (`,;:`) past 24 chars, with a word-boundary hard cap at 2× that — the speaker starts after a clause, not a sentence (unit-tested; note the cap only triggers under *streamed* pushes — a single large `push` finds sentence boundaries first). (3) **Barge-in**: `SpeakerPlayback::clear()` (flume is MPMC — a second receiver drains the queue out from under the device callback) + the drain loop now monitors mic RMS (>0.035 sustained 200 ms → cut playback, resume listening; threshold is high because there is **no echo cancellation** — speaker bleed must not self-trigger). `Turn` gained `ttft_ms`/`first_audio_ms`; both the live loop and `converse --input` print the breakdown (10.5). Hangover tightened 600→400 ms in the live loop. **Remaining floor: Kokoro RTF ≈ 0.66 on the first fragment** (~1.5–2.4 s) — next levers are a smaller first fragment or Kokoro acceleration; sub-1 s needs one of them.

### EnergyVad tuning (the "stuck 30 s" fix)
`VadConfig` gained `min_rms` (0.01 — absolute speech floor so room ambient ~0.003–0.01 isn't mistaken for speech; the old `1e-4` floor made every utterance run to `max_utterance_frames` = 30 s) and `end_ratio` (0.35 — a relative-drop detector: in the speech state a frame ends the turn when its RMS falls below `end_ratio × decaying-peak`, so a real pause finalizes even when ambient sits above `min_rms`). Tested in `vad.rs` (`ambient_below_min_rms_never_triggers`, `finalizes_promptly_on_real_pause`).
### `sapient stats` (live resource monitor)
`stats.rs` (`Commands::Stats`, aliases `top`/`monitor`): a ~1 Hz in-place TUI via the `sysinfo` crate — aggregates all `sapient`-named processes' CPU% + RSS, per-core CPU bars (flags the hottest core), system memory, and on-disk footprint (HF model cache via `hub::hub_cache_size()` + binary size). **Accelerator section is binary-gated** (`accel_label()` keys off `cfg!(feature="mlx"/"wgpu")`): a GPU build shows the GPU(s) + memory (via `sapient_generate::detect_devices().gpus`); a CPU build shows "accelerator none (CPU-only build)". Live GPU% isn't sampled (no portable API); on unified-memory Apple GPUs the model footprint shows in the process RSS. Ctrl-C to exit.
### Streaming TTS (real-time-ready audio path)
`SpeakPipeline::speak_streaming` / the `Tts::synthesize_streaming` override emit audio **as the codec LM decodes** instead of after the whole clip: `Pipeline::generate_token_ids_streaming` yields each token to a callback; per ~8 frames (`STREAM_CHUNK_FRAMES`) `emit_stable` re-decodes the running SNAC sequence and emits only the **stable prefix**, holding back `STREAM_MARGIN_FRAMES` (8) of look-ahead so an already-played sample never changes when more right-context arrives (SNAC convs have a finite right receptive field). `converse`'s `respond_streaming` calls `synthesize_streaming` per sentence → `SpeakerPlayback.submit`, so playback starts before the sentence finishes. Validated byte-exact vs batch decode by `tests/snac_coherence.rs::decode_prefix_is_stable` (max_err 0 over a 28-frame stable prefix with margin 8). **Caveat:** streaming lowers latency-to-first-audio but can't beat the LM's throughput. Two model-free speedups landed: (1) the **sampler** no longer full-sorts the vocab — `top_p_filter` partitions to the top-1024 via `select_nth_unstable` then sorts only those (Orpheus's 156 940-vocab `top_p` was sorting 156 k logits *per token*); this lifted Orpheus decode **~12 → ~17.5 codec tok/s on Metal (~1.4×)**. (2) a **pre-roll buffer** in `converse` (`PREROLL_SAMPLES` ≈ 0.6 s) holds the first audio so brief decode dips don't underrun the speaker. Even so, ~17 tok/s is still below the ~82 SNAC needs for glitch-free real-time. **The real-time enabler shipped: Kokoro-82M** (non-autoregressive, RTF ≈ 0.79 on M4 CPU — see its section). This streaming infra remains for the Orpheus path and any future autoregressive codec TTS.
### Orpheus text-to-speech (audio path, `sapient speak`) — 6d, LM-codec TTS
On-device TTS reuses the **text** `LlamaForward` engine + a pure-Rust neural-codec decoder; no new forward engine, no G2P/espeak dep. `sapient speak <model> "<text>" -o out.wav [--voice tara]`.
- **Two stages.** (1) **Orpheus-3B** (a Llama-3.2-3B fine-tune, registry alias `orpheus-3b` → `isaiahbjork/orpheus-3b-0.1-ft-Q4_K_M-GGUF`, runs on `LlamaForward`/GGUF) emits SNAC audio-codec **token ids**, not text. (2) the **`SnacDecoder`** (`forward/snac.rs`) turns those into a 24 kHz waveform; `sapient-audio::write_wav` writes it.
- **`SpeakPipeline`** (`sapient-generate/src/speak.rs`): loads the LM via `Pipeline`, the codec via `load_snac`, then `speak()` = build prompt → `Pipeline::generate_token_ids` → filter/de-frame → `SnacDecoder::decode` → waveform. Runs synchronously (CPU-bound); the CLI calls it in `spawn_blocking`.
- **`sapient speak` plays the audio by default.** `speak_command`/`speak_kokoro_command` synthesize the samples **once** (`SpeakPipeline::speak` / `KokoroTts::synthesize`, not `speak_to_wav` — synthesizing twice would re-run a slow 3B Orpheus LM), then `write_and_play_speech` writes the WAV **and** plays it through `SpeakerPlayback::default_output` (drain-then-250 ms-tail wait, mirroring `converse`). `--no-play` writes only. Playback is gated on the `audio-io` feature (a `--no-default-features` build prints a "saved to file only" hint); a headless box with no output device is a soft failure (WAV still written). `write_wav` is re-exported unconditionally from `sapient-generate` for this.
- **`Pipeline::generate_token_ids(prompt_ids, max_new, stop_ids, strategy)`** — the raw-token-id generation path (no detokenize), reused from the loaded engine. Sampling for Orpheus is **`Combined { top_p: 0.9, temperature: 0.6, repetition_penalty: 1.1 }`** (greedy/rep<1.1 degenerates to babble).
- **Prompt format (critical, must match the reference exactly):** `[128259] + tokenizer("{voice}: {text}") + [128009, 128260, 128261, 128257]`, stop at `128258`. The body is encoded **with special tokens** (`encode_ids(.., true)`) so the Llama-3.2 post-processor prepends **BOS `128000`** → realized prefix `[128259, 128000, …]`. **Dropping the BOS yields fluent-but-wrong speech** (verified: round-trip transcript was garbage without it). `SapientTokenizer::encode_ids(text, add_special)` is the raw encode that skips SAPIENT's own BOS injection.
- **De-framing** (`orpheus_codes_to_snac`): Orpheus packs **7 codes per frame**, each offset by position (`code = id − 128266 − pos·4096`); positions `0`→L1, `{1,4}`→L2, `{2,3,5,6}`→L3 (SNAC `vq_strides = [4,2,1]`). `speak()` keeps ids in `[128266, 128266+28672)`, rebases by `−128266`, truncates to a multiple of 7, then de-frames.
- **`SnacDecoder`** is fully convolutional (RVQ `from_codes`: codebook embed → `out_proj` 1×1 → repeat_interleave → sum; then conv `Decoder`: depthwise conv-in → per-rate `DecoderBlock` [Snake → `ConvTranspose1d` → 3× dilated `ResidualUnit`] → Snake → conv-out → tanh). The learned **NoiseBlock is omitted** (it injects `randn` — stochastic; the deterministic path is what we validate/ship). conv primitives in `forward/conv.rs` (`conv1d`, `conv_transpose1d`, `snake`).
- **SNAC weight source + `normalize_snac_weights`:** the released codec ships only a `.pth` pickle, so SAPIENT pulls the ungated **`mlx-community/snac_24khz`** safetensors mirror (`HubClient::download_files`, a raw repo-id download that bypasses the registry). That mirror stores **un-folded** weight_norm (`*.weight_g`/`*.weight_v`), **MLX channel-last conv layout `[d0,K,d1]`**, and a `…layers.N…` key prefix; `normalize_snac_weights` folds the weight_norm, swaps each conv kernel's last two axes to PyTorch `[d0,d1,K]`, strips `.layers.`, and drops encoder-only `in_proj.*`. Already-folded torch weights (from `scripts/convert_snac_to_safetensors.py`) pass through unchanged. Override the weights with a local dir via `SAPIENT_SNAC_DIR`.
- **`latent_dim` gotcha:** the `mlx-community` `config.json` **omits `latent_dim`** (the torch one has `768`). `SnacDecoder::decode` derives the continuous-latent width from the conv-in weight `decoder.model.0.weight` shape rather than mis-defaulting to `codebook_dim` (8) — defaulting to 8 silently produces garbage audio of the right length.
- **Tokenizer fallback:** Orpheus's GGUF `general.name` is "Orpheus Tts 0.1 Pretrained"; `tokenizer_fallback_model` maps any id containing `orpheus` → ungated **`unsloth/orpheus-3b-0.1-ft`** (vocab **156940** — the base Llama-3 tokenizer at 128256 would fail the vocab-compat check).
- **Validation:** `tests/snac_coherence.rs` (ignored, `$SAPIENT_SNAC_DIR`) decodes the committed fixture bit-close to a torch reference (max_err ~2e-6) for **both** the `mlx-community` and converted weight layouts. End-to-end gate is the **speak→transcribe round-trip** with the validated Whisper STT ("Hello there." → speech → "Hello there.").

### Kokoro-82M text-to-speech (audio path, `sapient speak kokoro-82m`) — the real-time TTS
Orpheus-3B is autoregressive (one codec token per forward pass) → ~0.18× real-time even on Metal, too slow for live `converse`. **Kokoro-82M** is a *non-autoregressive* StyleTTS2 + ISTFTNet model: **one forward pass** turns phonemes + a voice style vector into a 24 kHz waveform, no codec-token loop — so it sidesteps the tokens/sec ceiling entirely. Pure-Rust port lives in `sapient-models/src/forward/kokoro/` (CPU-only, ~82M params); **measured RTF ≈ 0.48 on M4 CPU after the parallel-conv work (was 0.79 at ship, 0.66 pre-v0.5.2-kernels)**, ~12× faster than Orpheus for a comparable reply. Speak→transcribe round-trip is exact ("Hello there. How are you today?" → Whisper → same).
- **Five modules** (key prefixes match the checkpoint): `bert` (PLBERT = HF **ALBERT**, one shared `albert_layer` run 12×, factorized `embedding_size=128`→768, learned pos, `gelu_new`) → `bert_encoder` (Linear 768→512) → `predictor` (ProsodyPredictor: bidirectional LSTMs + AdaLayerNorm → per-token durations; shared LSTM + AdaIN+LeakyReLU `AdainResBlk1d` stacks → F0/N curves) → `text_encoder` (conv k5 + channel-LayerNorm + LeakyReLU + BiLSTM) → `decoder` (ISTFTNet: AdaIN+Snake `AdaINResBlock1`, ConvTranspose upsampling ×10×6, NSF harmonic source injected into the conv stream, `conv_post` → exp/sin → iSTFT). Durations build a hard alignment that length-regulates `en` and `asr` (segment-repeat = `length_regulate`). Style is `ref_s[256]` split **128/128** (decoder timbre `[:128]` / predictor prosody `[128:]`), indexed by phoneme count (`pack[len(ps)-1]`).
- **New CPU kernels** (all in `kokoro/ops.rs`, each unit-tested vs a reference, since SAPIENT lacked them): bidirectional **LSTM** (`lstm_bidirectional`, PyTorch i,f,g,o gate order), **STFT/iSTFT** (`stft_transform`/`istft`, `torch.istft`-equivalent — irfft **1,2,1** bin weighting, periodic Hann, **window² overlap-add** normalisation, center-trim `n_fft/2` each end → `hop·(T−1)`; matches the shipped `TorchSTFT`, *not* the approximate ONNX `CustomSTFT`), **AdaLayerNorm**/**AdaIN1d** (LayerNorm/InstanceNorm1d + style γ/β; `InstanceNorm1d(affine=True)` in source but no affine params in the checkpoint → affine-free is correct), **length regulator**, depthwise ×2 transposed-conv `convtr_depthwise_up2` (the upsampling `AdainResBlk1d.pool`, `output_padding=1`), and the **NSF harmonic source** (`nsf_harmonic_source`: 9 harmonics, `sine_amp=0.1`, `f0>10` voiced, upsample 300). conv1d/conv_transpose1d/snake/leaky_relu are reused from `forward/conv.rs`.
- **Determinism:** the NSF source omits the reference's training-time Gaussian noise + random initial phase (negligible — voiced noise std 0.003, rand-phase is first-sample-only) → reproducible. Final audio is therefore spectrally identical but not bit-exact (raw-sample correlation is meaningless for speech with differing excitation phase) — same precedent as omitting SNAC's NoiseBlock. Coherence gate uses the **energy envelope** (0.999) + log-magnitude spectrogram (0.99), not max_err.
- **Performance:** ALBERT's big projections + the decoder convs route through SAPIENT's SIMD+rayon `matmul_nt` (`linear_mm`) instead of naive loops (ALBERT 1013→74 ms); the decoder's 3 independent resblocks per upsample stage run across cores via `rayon`. This is what gets it under 1.0× RTF.
- **Weight source:** upstream `hexgrad/Kokoro-82M` ships only a `.pth` pickle (+ `.pt` voice packs), so — like SNAC — `scripts/convert_kokoro_to_safetensors.py` converts **once, offline** (flatten the 5 per-module sub-dicts, strip `module.`, voices `[510,1,256]`→`[510,256]`) to safetensors hosted at the mirror **`sai1974dev/kokoro-82m-safetensors`** (`KOKORO_REPO`). The loader (`kokoro/loader.rs`) folds `weight_g`/`weight_v` into `.weight` (reuses SNAC's `weight_norm_fold`). Override with a local dir via **`SAPIENT_KOKORO_DIR`**.
- **Input length cap / id bounds (panic guard):** `phonemes_to_ids` truncates the phoneme sequence to ALBERT's learned **position-embedding capacity** (`position_embeddings` rows − 2 pads, ~510) and drops any id ≥ the embedding row count. Without this, a long LLM reply (e.g. a runaway repetitive generation) overflowed the position embedding and a mirror's `vocab` mapping a rare IPA symbol to an out-of-range id overflowed `word_embeddings` — both panicked mid-synthesis with `index out of bounds` (in `kokoro/albert.rs`). The two embedding loops (`albert_encode`, `text_encode`) also defensively clamp the id and position so the `pub synthesize_ids` path can't panic on raw ids. The reference Kokoro chunks long text into ≤510-phoneme segments; truncation is the minimal guard (a future improvement is sentence-chunked synthesis + concat).
- **G2P:** `KokoroTts` (`sapient-generate/src/kokoro_tts.rs`) implements the `Tts` trait; text→phonemes via the pure-Rust **`misaki-rs`** crate (`default-features = false`, no espeak-ng/GPLv3), the same G2P family Kokoro was trained with. `KokoroTts::from_default()` pulls `KOKORO_REPO` (or `SAPIENT_KOKORO_DIR`); `from_dir` loads a local dir. `sapient speak kokoro-82m "<text>" [--voice af_heart]` routes here (CLI `is_kokoro_model`); **`converse --speak` now uses Kokoro** (real-time) instead of Orpheus.
- **Validation:** `kokoro/ops.rs` unit tests (9, non-ignored); `kokoro/stage_tests.rs` (ignored, `$SAPIENT_KOKORO_DIR` + committed fixture `tests/fixtures/kokoro_hello.safetensors`) checks **every stage** against a PyTorch reference — ALBERT 1e-5, bert_encoder 8e-7, durations exact, en 2e-6, text_encoder 1.6e-6, F0 1.5e-4, audio envelope 0.999; `sapient-generate/tests/kokoro_tts_e2e.rs` (ignored) is the text→audio + RTF + `from_pretrained`-download path. Fixtures are regenerated from the local kokoro source (no kokoro/misaki pip dep) — see the test headers.

### Gemma3 engine (`Gemma3Forward`) — gemma-3-1b/4b + MedGemma
CPU text engine for the Gemma3 family (`forward/gemma3.rs`), plus the Gemma3 **multimodal** path in `VlmPipeline` (gemma-3-4b-it, **MedGemma-4B** — medical image analysis via `sapient see scan.png --model medgemma-4b`; gated repo, needs `sapient login`). The five Gemma3-vs-Llama differences and their handling: (1) **zero-centered RMSNorm** `x/rms·(1+w)` — the `+1` is folded into every norm weight ONCE at load (`fold_gemma_norms`, incl. the per-head **QK-norms**), after which everything is a standard `rms_norm` call; (2) **sandwich norms** (post-attention/post-FFN norms applied to the branch OUTPUT before the residual — 4 norms/layer); (3) **alternating attention**: `(idx+1) % sliding_window_pattern != 0` → sliding-window mask, else global; local layers RoPE at `rope_local_base_freq` 10k, global at `rope_theta` 1M with optional linear pos-scaling (`apply_rope_partial_scaled`, factor 8 on 4B+); (4) explicit `head_dim` (256 ≠ hidden/heads) and attention scale `query_pre_attn_scalar^-0.5`; (5) **√hidden embedding scale** and a tied 262k-vocab embedding quantized to Q8_0 at load (gather + logits GEMV). **Two hard-won traps:** `"Gemma3ForCausalLM".contains("Gemma")` matched the old `Gemma` arch arm first and silently routed to the Llama engine → token salad — `Gemma3` must match before the catch-all in `from_hf_arch_name`; and the **flash-attention NaN kernel bug**: `flash_attn_row` computed `exp(-inf − (−inf)) = NaN` when the FIRST attended position was masked — unreachable from causal decode, first hit when a sliding window scrolled past position `sliding_window` (Gemma3 was coherent for ~490 tokens then salad at ~512; masked positions now `continue`, regression test `leading_masked_positions_do_not_nan`). Multimodal: the SigLIP tower is shared with SmolVLM (`SiglipVision::with_prefix("vision_tower.vision_model")`, `encode_features` = raw tower output) + the Gemma3 connector (`gemma3_project` in vlm.rs: 4×4 avg-pool 4096→256 tokens → zero-centered RMSNorm → raw matmul with `mm_input_projection_weight`); prompt protocol `<bos><start_of_turn>user\n\n\n<start_of_image>` + `<image_soft_token>`×256 + `<end_of_image>\n\n{q}<end_of_turn>\n<start_of_turn>model\n`, image token id 262144. Tokenizer-compat: a tokenizer up to 64 tokens LARGER than model vocab is allowed (Gemma3 ships `<image_soft_token>` beyond the text vocab). Sharded safetensors handled in `VlmPipeline::from_pretrained` (index → shard list).

### SmolVLM vision-language (`sapient see`) — Phase 12 first cut
`VlmPipeline` (`sapient-generate/src/vlm.rs`) runs the Idefics3/SmolVLM family: a **SigLIP vision tower + pixel-shuffle connector** (`forward/siglip.rs`, CPU f32 — patch-embed via the parallel `conv2d`, pre-LN blocks with the Whisper-style **explicit zeros mask** (never `None` — that means causal), `gelu` = the tanh approx SigLIP wants, NOT `gelu_erf`) feeding a **stock `LlamaForward`** text backbone (SmolVLM-256M's is SmolLM2-135M). The one new text-engine mechanism is **embedding splicing**: `LlamaForward::token_embeddings` + `forward_logits_embeds` let the pipeline overwrite the `<image>` rows with visual embeddings and prefill from embeddings; decode steps are the normal token-id path on the same KV cache. **Critical details:** (1) the Idefics3 **pixel-shuffle ordering** is `out[hj][wj] = concat_{dh outer, dw inner} in[hj·s+dh][wj·s+dw]` (unit-tested — this model's token-salad-class bug); (2) the **prompt protocol** is `<|im_start|>User:<fake_token_around_image><global-img><image>×64<fake_token_around_image>{q}<end_of_utterance>\nAssistant:` with `add_special=false` (the processor-level image expansion is done by the pipeline); (3) checkpoint keys split by prefix — `model.text_model.*` → `model.*` for the Llama engine, vision/connector keys kept verbatim; (4) v1 scope: single squash-resized 512² global image (no Idefics3 sub-image splitting), mean/std 0.5. **Validation:** `vlm_e2e` golden test (solid-red fixture → "Red"), and `vlm_geometry_probe` verifies grid orientation NUMERICALLY (vertical split → column-dominant token norms, horizontal → row-dominant) because the 256M LM's spatial words are too unreliable to test geometry with. Registry: family `SmolVLM` → `ModelCategory::Vision`, CLI `sapient see <image> -p "…" [--model smolvlm-256m]`. **MedGemma is NOT in the catalog:** it's Gemma3 (QK-norm, alternating sliding/global attention, pre+post-FFN norms) — needs a new text engine first; a catalog row that can't load violates the registry rule.

### Kokoro streaming: measured NO-GO on windowed decode (gate 3)
Three streaming designs were built and measured (duplex-spike gate 3): growing-prefix (RTF 1.55 → underruns; prefix chains can't be gap-free at decoder-RTF 0.53), rolling windows with halo (`KokoroModel::decode_window` + `nsf_harmonic_source_from` analytic phase carry — bookkeeping byte-exact, but **AdaIN/InstanceNorm time statistics are global**: windows diverge 0.45/0.34/0.20 max-diff at halo 16/32/64, `kokoro_window_probe` is the regression record), and a geometric text ramp (first audio 5.2 s → 1.0 s but the speak→transcribe round-trip catches sentence-final prosody at every join — audibly wrong). **Kokoro therefore uses the batch `Tts::synthesize_streaming` default; converse's clause-level chunker is the right granularity.** Stage profile of a first-clause fragment: decoder = 76% (1332 ms of 1741 on M4) — the TTFA floor is decoder RTF. **The first kernel rung landed:** per-op counters (im2col/GEMM/conv_transpose/snake/adain, printed under `SAPIENT_KOKORO_TIMING`) convicted the conv GEMM (1437 ms thread-summed — `conv2d` ran ONE single-threaded `matrixmultiply::sgemm` per conv). Parallelizing the GEMM across output-channel row blocks (disjoint sgemm calls — bit-identical) + parallel row-wise im2col + parallel snake: decoder 1353 → **840 ms**, Kokoro RTF 0.66 → **0.48**, converse first-audio 2724 → **2013 ms**; all 5 PyTorch-reference stage gates + round-trip pass. (Nested vs flat resblock parallelism A/B'd: nested wins.) Every conv2d consumer (Whisper encode, SNAC, prosody) inherits the parallel kernels. Remaining paths: GEMM blocking/precision work, then the streaming-norm redesign. The `decode_window`/phase-carry infra stays (validated, tested) for that future.

### MlxForwardEngine — critical invariants
- **RoPE axis:** `mlx_rs::fast::rope` uses dimension **−2** as the sequence-position axis. q/k/v MUST be transposed to `[1, n_heads, seq, head_dim]` (seq at −2) **before** RoPE — exactly like mlx-lm. Applying RoPE to `[1, seq, n_heads, head_dim]` scrambles positions across heads and produces garbage (every model collapses to one repeated token). This was the v0.3.4 fix.
- **KV cache** is stored in `[1, n_kv_heads, seq, head_dim]` layout and grown via `concatenate_axis` on axis 2.
- **GQA:** MLX's fused `scaled_dot_product_attention` handles grouped-query attention correctly (K/V not pre-tiled) — *once RoPE is on the right axis*. The earlier "fused SDPA mishandles GQA" symptom was the RoPE bug; with that fixed, the fused Metal kernel is both correct and faster than a manual matmul loop, so the engine uses it for all attention.
- **Engine reuse:** `Pipeline.engine` is `Arc<Mutex<ForwardEngine>>`. The streaming generators (`generate_stream`, `generate_stream_with_config`) clone the Arc and reuse the loaded engine — they must NOT rebuild it via `ForwardEngine::from_weight_paths_with_backend`, which re-quantizes the whole model and dominated TTFT (3 s on 1.5B). The non-streaming `generate_from_tokens_with_config` uses `tokio::task::block_in_place` on `self.engine.lock()`.

### WgpuForwardEngine — critical invariants (`feature = "wgpu"`)
- **Layout / kernels:** all kernels live in `sapient-backends-wgpu` `resident.rs` + `quant.rs` (host side) + `shaders/*.wgsl`. Convention: WGSL entry point is always `cs_main`; **f32 accumulation everywhere** (f16 accumulation is incoherent — deep-research finding); dispatch grids are 2-D-tiled (`idx = wg.x + wg.y*num_workgroups.x`) so counts above the 65535 per-dimension limit (e.g. a 152k-row `lm_head`) still launch — `WgpuContext::dispatch` does `gx = groups.clamp(1,65535); gy = groups.div_ceil(gx)`.
- **RoPE convention:** `rope.wgsl` is in-place NEOX rotate_half over the first `rotary_dim` channels (`rotary_dim = partial_rotary_factor·head_dim`, rounded even), `freq = pos/base^(2i/rotary_dim)` — must match the CPU `apply_rope_partial` exactly (same class of axis bug as MLX would scramble heads). Data is `[rows, head_dim]` where `row % seq_len` is the position index.
- **Attention:** `attention.wgsl` is causal GQA FlashDecoding (online softmax). It takes a **`kv_stride`** distinct from `seq_k`: the KV cache is pre-allocated `[n_kv_heads, max_seq, head_dim]` (stride `max_seq`) but only the first `seq_k` positions are valid — `kv_base` uses `kv_stride`, the attend bound uses `seq_k`. One workgroup per `(batch, head, query-row)`, 128 lanes parallel over `head_dim` (`jcount = ceil(head_dim/128)`, register arrays size 4). GQA is handled in-kernel via `kv_rep = n_heads/n_kv_heads` — K/V are **not** pre-tiled.
- **KV-cache append:** `copy_range` is a pure `copy_buffer_to_buffer` (no shader, no readback). The engine appends a freshly-computed K/V head-slice into its cache slot per kv-head per decode step — the cache never leaves the GPU.
- **Q8_0 weights stay quantized on the GPU (Phase 7.1).** `upload_q8_0` repacks raw ggml Q8_0 blocks (34 bytes: LE f16 scale + 32 int8) into a `GpuQ8Buffer` — int8 quants packed 4-per-`u32` word (little-endian lanes) + scales widened once to f32 — with **no host-side f32 expansion**; `matmul_nt_q8_0.wgsl` / `embed_q8_0.wgsl` dequantize in-shader (`f32(scale_f16)·f32(int8)`, exactly the CPU's dequant values). ~1.125 bytes/weight of VRAM vs 4 for f32 (SmolLM2-360M Q8_0: 1.6 GiB → 388 MiB ≈ GGUF file size; peak RSS 2.65 → 1.27 GB). The engine's `upload_weight` keeps `DType::Q8_0` tensors quantized, **online-quantizes F16/BF16 linears via the same `should_quantize_online` rule as the CPU engine** (so both engines see identical weight values), and stores **tied output projections as `lm_head: None`** — logits reuse the embed buffer instead of uploading the matrix twice. Fallback for everything else (F32, Q5_K/Q6_K until they get shaders, rows not a block multiple) is f32 upload. Greedy decode of a real Q8_0 GGUF is token-identical to the old f32 path (CPU may legitimately diverge after tens of tokens — its aarch64 W8A8 SDOT path quantizes *activations*; the GPU keeps them f32).
- **Q4_K weights stay quantized too (Phase 7.2).** `upload_q4_k` uploads raw 144-byte super-blocks **verbatim** (144 % 4 == 0 → binds directly as `array<u32>`, zero repack, zero dequant): word 0 = `unpack2x16float` → (d, dmin); words 1–3 = the 12 packed 6-bit scale/min bytes (in-shader `get_scale_min_k4`, the same +4/high-bit packing as the CPU `kernels/quant.rs` — this is the Q6_K-postmortem bug class, change nothing without the random-bit reference tests); words 4–35 = qs nibbles (sub-block `is = e/32`, byte `(is/2)*32 + e%32`, low nibble for even `is`). Per sub-block the shader accumulates `Σx·q` and `Σx` then applies `d·sc`/`dmin·mn` once. 0.5625 bytes/weight; gated on `k % 256 == 0`. **Coherence-test design note:** the K-quant gate compares GPU-quantized vs CPU running a *host-dequantized f32 twin* (`to_f32_vec`) — tight 5e-3 bound because both sides then use identical weight values and f32 activations; comparing vs the CPU quantized path instead adds ~0.1 of W4A8 activation-quant noise that would mask real dequant bugs.
- **Q6_K completes Q4_K_M coverage.** ggml Q6_K blocks are **210 bytes — not word-aligned** — so `upload_q6_k` pads each to 212 (memcpy only): words 0–31 = ql[128], 32–47 = qh[64], 48–51 = 16 **signed** int8 scales, word 52 = f16 d. Element `e` (h=e/128, g=(e%128)/32, l=e%32): `q = ((ql[h·64+(g&1)·32+l] >> (g/2)·4) & 0xF) | (((qh[h·32+l] >> 2g) & 3) << 4)`, `val = d·sc[h·8+2g+l/16]·(q−32)` — the +0/+2/+4/+6 per-half scale indexing mirrored from the *fixed* `dequantize_row_q6_K` (this exact indexing was the historical token-salad bug). Result on Qwen2.5-1.5B Q4_K_M (M4 16 GB): **198/198 matrices quantized**, weights resident 6778 (f32) → 2367 (Q4_K only) → **1062 MiB** (≈ GGUF file size), peak footprint 14.7 → **3.6 GB**, decode 11.3 → **13.2 tok/s (1.13× the M4 CPU path)** — on 16 GB the f32 path had exhausted memory and emitted immediate-EOS empty replies. Coherence: the mixed-quant tiny-Llama uses Q6_K exactly where Q4_K_M files do (v_proj + lm_head).
- **f16 KV cache (Phase 7.3) — packed u32, not `enable f16`.** naga in wgpu 22 does **not** parse `enable f16;`, so the f16 cache stores halves packed two-per-`u32` word: `kv_append_f16.wgsl` writes with `pack2x16float` (one thread per word — head_dim must be even so a word never straddles a head; that's the only requirement, no device feature), and the attention f16 variant is generated by `f16_kv_variant()` rewriting the `k_at`/`v_at` accessor lines of `attention.wgsl` to `unpack2x16float` + `select` (labels `attention` vs `attention_f16kv` in the pipeline cache). Accumulation stays f32. Auto-on when `head_dim % 2 == 0`; `from_weights_with_kv(.., Some(false))` forces f32 (the tight coherence gates use this). Half the KV bytes lifts the wgpu ctx cap 4096 (`WGPU_MAX_CTX_F32`, f32 fallback only) → the standard `kv_cache_ctx` 8192. `kv_append` also replaced the per-kv-head `copy_range` loop on the f32 path; `copy_range` itself remains (Whisper wgpu uses it).
- **Per-token command batching (Phase 7.4).** `WgpuContext::begin_batch()`/`flush_batch()`: while a batch is open, `dispatch()`/`copy_range()` record into one shared command encoder instead of one queue submission per kernel; `download_f32` flushes automatically. `forward_token` opens a batch and **flushes at its own end — never let a whole prompt accumulate**: prompt_len × ~450 compute passes in one encoder stalls Metal outright (each pass is a Metal command encoder in a single command buffer); the tiny coherence models never trigger this, only real models do. Measured: SmolLM2-360M +27% decode, Qwen-1.5B +4% (M4/Metal). Immediate mode (no open batch) remains for tests + Whisper wgpu; equivalence pinned by `batched_recording_matches_immediate`.
- **Batched prefill (Phase 7.5).** `forward_logits` chunks prompts at `PREFILL_CHUNK = 128` through `forward_chunk`; single tokens use the transpose-free decode path. The layout dance is the invariant to know: matmuls produce **seq-major** `[seq, heads, head_dim]`, while RoPE (position = `row % seq`), the KV cache, and the attention kernel's q are **heads-major** `[heads, seq, head_dim]` — q/k/v are `transpose_heads`-ed after the projections, and the attention output transposes back before o_proj (with `seq_q = 1` the layouts coincide, which is why decode needs no transposes — and why the bug wouldn't show in single-token tests). `kv_append` takes a `seq` param (`[n_kv, seq, head_dim]` heads-major src → positions `pos..pos+seq`); biases use the broadcast `add_bias` (decode's equal-length `add` doesn't broadcast over rows). Gated by `wgpu_chunked_prefill_matches_per_token` (300 tokens: two full chunks + partial + pos0>0). **Known limitation:** the matmul kernels are GEMV-shaped, so chunking doesn't yet amortise weight reads (measured 1.5× TTFT on a 640-token prompt, not chunk×) — multi-row/tiled GEMM is the follow-up.
- **Scope:** Llama-family only (RMSNorm, SwiGLU, full RoPE, optional q/k/v bias). P5-remaining: multi-row GEMM for prefill, scratch-buffer/bind-group reuse; shader fusion was evaluated and deferred (see ROADMAP).
- **Coherence is the gate:** `crates/sapient-models/tests/wgpu_coherence.rs` builds a synthetic tiny Llama and asserts `WgpuForwardEngine` logits match the CPU `LlamaForward` (prompt + incremental decode, argmax + `max_err < 5e-3`), plus a **Q8_0 variant** (`wgpu_q8_0_logits_match_cpu_llama`: every 2-D weight as raw Q8_0 blocks, tied lm_head; argmax equality + bounded err — looser because of the CPU's W8A8 activation quant). Use `head_dim` not divisible by 32 so the CPU reference also keeps an F32 KV cache (clean f32-vs-f32 compare). Per-kernel CPU-reference tests are in `sapient-backends-wgpu/tests/resident.rs` (Q8_0 GEMV/embed vs exact dequant reference). I can only validate Mac→Metal locally; the same WGSL must be CI-validated on real Vulkan/DX12 targets.

### Quantized storage (Phase 1)
`DType::Q4_0` and `DType::Q8_0` store raw ggml block bytes — no F32 expansion at load time. Key invariant: `as_bytes()` on non-quantized tensors returns the full buffer from `offset`; on quantized tensors it returns exactly `byte_count(numel)` bytes. Use `as_quant_blocks()` to access raw blocks, and `to_f32_vec()` to dequantize. `matmul_nt` dispatches on weight dtype: float weights use SGEMM, quant weights use per-block dot products.

### GGUF loading (Phase 4: mmap)
Three loading paths in `sapient-io/src/gguf.rs`: `parse_metadata_only` (header KV only, zero tensor alloc), `load_tensors_with_metadata` (heap CpuBuffer), `load_tensors_mmap` (OS-managed paging via `MmapBuffer` — Q4_0/Q8_0 zero-copy, K-quants dequantized from mmap bytes). GGUF dims are ggml column-major `[in, out]`; `map_gguf_tensors_to_hf` flips to HF `[out, in]`. The pipeline auto-detects mmap when file > 80% of available RAM (`available_ram_bytes()` reads `/proc/meminfo` on Linux, `sysctl` on macOS). `--mmap` flag forces it; `Pipeline::is_mmap()` reports which path was taken.

### GGUF q/k RoPE permutation (critical for llama-arch GGUF)
llama.cpp's HF→GGUF converter **permutes the rows of `attn_q`/`attn_k`** for the `llama` architecture (Llama, Mistral, SmolLM, TinyLlama, …) because ggml uses NORM-style RoPE while HF/SAPIENT use NEOX-style (`rotate_half`). Loading those weights as-is makes RoPE scramble positions across each head → **incoherent token-salad output** (correct config, correct shapes, finite activations — just wrong). `gguf_weights::unpermute_llama_gguf_qk` (called from `ForwardEngine::from_gguf_weights` **only** for `ArchType::Llama`) inverts the permutation: HF row `h·D + a·(D/2) + b` ← GGUF row `h·D + b·2 + a`. It runs on any dtype (each output row is a contiguous `byte_count(in)` chunk). **Qwen2/Gemma GGUFs use NEOX RoPE — NOT permuted by the converter, so they must NOT be un-permuted** (the gate excludes them). Safetensors models are already HF layout. Regression test: `unpermute_qk_inverts_llama_cpp_permutation` in `gguf_weights.rs`.

### Tied-embedding GGUF models
GGUF metadata has no `tie_word_embeddings` flag, so models that tie the output projection to the input embedding (SmolLM2, Llama-3.2-1B/3B, small Qwen) simply omit `output.weight`. `resolve_lm_head` (in `weights.rs`) falls back to the embedding matrix when no explicit `lm_head`/`output` weight exists — otherwise those models fail to load with "missing lm_head.weight".

### KV-cache context window cap (OOM guard)
The KV cache is pre-allocated for `max_seq` positions up front. Modern models advertise huge contexts (Llama-3.1/DeepSeek-R1 = 131072) → ~9 GB of Q8_0 cache for an 8B model, OOM-killing a 16 GB device at **load** time. `common::kv_cache_ctx(model_max)` caps the allocation to `DEFAULT_KV_CACHE_CTX` (8192), overridable via the `SAPIENT_CTX` env var; longer conversations slide the window. Both `LlamaForward` and `PhiForward` cap `LayerCache.seq_len` to the allocated window so the sliding-window `update_kv_cache` never indexes past the cache.

### GGUF quant-file selection & tokenizer fallback
`sapient-hub/src/gguf.rs::gguf_preference_score` ranks GGUF files; **Q4_K_M is preferred over Q8_0** (edge sweet spot — ~40% smaller, near-lossless, uses the Q4_K matmul kernel; the old code picked Q8_0 and shipped an 8.5 GB file where a 4.9 GB one fits a 16 GB Pi). `tokenizer_fallback_model` resolves a HF tokenizer repo for GGUF-only models: it must point at **ungated** repos AND the **right version** — `meta-llama/*` returns 401 without a token, so Llama-3/DeepSeek fall back to `unsloth/Meta-Llama-3.1-8B-Instruct` and `deepseek-ai/DeepSeek-R1-Distill-Llama-8B`. **Mistral: the GGUF catalog ships Mistral-7B-Instruct-v0.3 (vocab 32768), which REORDERED the vocab vs v0.1/v0.2 (32000) — the tokenizers are NOT interchangeable.** Loading the v0.1 tokenizer for a v0.3 GGUF mis-encodes the prompt and mis-decodes output into mixed-script token-salad (verified: v0.1 decodes v0.3 ids as `Г str — …らíses…レ`). So `mistral` defaults to `unsloth/mistral-7b-instruct-v0.3`; only an explicit `v0.1`/`v0.2` in the id falls back to `mistralai/Mistral-7B-v0.1`. Regression test: `mistral_tokenizer_fallback_defaults_to_v03`.

### Q6_K dequantization scale indexing (critical)
Q6_K stores **16 int8 scales per 256-weight super-block** (one per 16-element group) + one fp16 super-scale. Dequant must map weight `i` to scale `i/16`: within each 128-element half the four sub-groups (offsets 0/32/64/96) use scale offsets **+0/+2/+4/+6** with a further split at `l==16` (`is = l/16`), and the scale base advances by **8** per 128-block (matching ggml `dequantize_row_q6_K`). The original code used one scale per 32-group (`sc[ib..ib+4]`, `ib += 4`) — it only ever touched scales 0–7 and decoded Q6_K **wrong**, corrupting any Q6_K tensor. This is fixed in **three** places that must stay in sync: `sapient-core` `Tensor::to_f32_vec` (embedding lookup / `to_f32_cow`), `sapient-backends-cpu` `dot_q6_k_row_f32` (matmul), and `sapient-io` `dequantize_q6_k`. Regression test: `q6_k_scale_indexing_matches_ggml` in `kernels/quant.rs`. This bug was why Q4_K_M models that use Q6_K for the output/embedding (Llama-3.x, DeepSeek-R1-Distill-Llama, Mistral Q4_K_M) emitted token-salad — especially catastrophic for **tied-embedding** models (Llama-3.2-1B/3B) where the Q6_K embedding is the model input. qwen models were unaffected (their embeddings are Q4_K; Q6_K appears only in v_proj, where the error was tolerable).

### Model registry (curated, not open)
`sapient-hub/src/registry.rs` contains a hardcoded `CATALOG` of `SupportedModel` entries. Every model resolves through `resolve_model_alias` — unrecognised names error with the catalog list. Fuzzy matching (prefix + Levenshtein) handles near-miss typos. Add a model by: (1) verify its arch is supported in a forward engine, (2) add a `SupportedModel` row with `openhorizon/*` branding.

**Capability category (`ModelCategory`).** `SupportedModel::category()` buckets each model by `family`: `Whisper`→`SpeechToText`, `Orpheus`/`Kokoro`→`TextToSpeech`, everything else→`Chat`. This drives two things: (1) `sapient models` prints **three grouped sections** (Text generation / Speech-to-text / Text-to-speech), each with its own run hint, instead of one flat table; (2) command validation — `speak_command` rejects a non-TTS model up front (e.g. `sapient speak whisper-small` → clear "is a speech-to-text model, use `sapient transcribe`" error, not the cryptic "architecture Whisper does not yet have a native forward engine"). Mirrors the existing `converse --stt` Whisper-family check. **Kokoro-82M is now a catalog entry** (`openhorizon/kokoro-82m` → `sai1974dev/kokoro-82m-safetensors`, family `Kokoro`) so it shows in `sapient models`; the CLI still routes it via `is_kokoro_model()` (it loads from the safetensors mirror, not the registry repo path).

### GGUF-only repos
When `from_pretrained` downloads a GGUF-only repo (no `config.json`), the hub client sets `config_path` to the GGUF file itself. The pipeline detects this via extension and routes to `from_gguf_opts`, bypassing `ModelInfo::from_config_file`.

### SIMD hot paths
`kernels/quant.rs` has four dispatch layers:
1. `dot_q4_0_block_f32` / `dot_q8_0_block_f32` — dispatch to NEON (`vfmaq_f32`) on `aarch64`, scalar otherwise.
2. `dot_q8_0_row_f32` — additionally dispatches to AVX2+FMA on `x86_64` with runtime `is_x86_feature_detected!`.
3. `dot_q4_k_block_f32` — NEON nibble-unpacking (`vshrq_n_u8` + `vandq_u8`) + `vfmaq_f32` FMA for Q4_K blocks.
4. Native F16 GEMV: F16 weights decoded in NEON registers via `vcvt_f32_f16`, no intermediate F32 allocation.

**Adaptive rayon chunking:** `gemv_chunk()` targets 4 tasks per core (not one task per output row). For a 151 936-row `lm_head` this avoids spawning 151 936 micro-tasks. `matmul_nt_q*` and `scaled_dot_product_attention` are parallelised with `rayon::par_iter_mut` / `par_chunks_mut` over the output dimension; `LlamaForward::forward_layer` uses `rayon::join` for parallel Q/K/V and gate/up projections.

**K-quant kernels are all NEON-vectorized (v0.3.9 — critical for RPi5/Cortex-A76 decode):** `dot_q4_k_row_q8_neon` (W4A8 SDOT — int8 activations via `quantize_row_to_i8_blocks` + `sdot`), `dot_q6_k_row_f32_neon`, and `dot_q5_k_row_f32_neon` (16-lane NEON). **Q6_K/Q5_K were previously pure scalar** while Q4_K was SIMD — and Q6_K is ~⅓ of a Q4_K_M model (lm_head + half of ffn_down + attn_v), so scalar Q6_K *dominated* decode and masked all Q4_K SIMD work. Vectorizing Q6_K gave +36–100% on the Pi (qwen-0.5B 4.5→6.1, 1.5B 1.3→1.9, mistral-7B 0.3→0.6 tok/s). Vectorizing Q5_K also surfaced+fixed a latent correctness bug: the scalar read the 5th bit as `qh[is/8]` (one bit per 32-element sub-block) instead of ggml's per-element `qh[l]` — would salad Q5_K_M models. Each NEON kernel is regression-tested bit-close to its scalar reference (`q6_k_neon_matches_scalar`, `q5_k_neon_matches_scalar`, `q4_k_w4a8_matches_f32_path`). **Lesson from the RPi5 perf hunt:** decode is memory-latency-bound, not compute-bound — SDOT, single-reduction, and multi-row/MLP kernels all gave ~0 once Q6_K was vectorized; the practical kernel ceiling is "no scalar K-quant kernels." Build the Pi binary on the Mac with `cargo zigbuild --release --target aarch64-unknown-linux-gnu` (rustls TLS means no openssl-sys cross-compile blocker); `RUSTFLAGS="-C target-cpu=cortex-a76"` is fine for the linux target (the ring/`target-cpu=native` caveat is only for `aarch64-apple-darwin`).

**Multi-row GEMV + R4 repacking + W6A8 + SMMLA (v0.5.x, the llama.cpp-gap work):** decode/prefill K-quant matmuls are multi-row. (1) `DType::Q4_K_R4`/`Q6_K_R4` are load-time **row-interleaved layouts** (groups of 4 rows, super-blocks block-major `[r0.b, r1.b, r2.b, r3.b, …]` — one contiguous weight stream per task): produced by `repack_q4_k_weights` in `forward/common.rs` for **pure-CPU aarch64+dotprod engines only** (heap tensors only — `Buffer::is_mmap` gates it; embeddings excluded and `gather_row_f32` hard-errors on R4; untied lm_head resolved after repack so it gets the layout; `SAPIENT_NO_REPACK=1` escape hatch; Q6_K repack is opt-in `SAPIENT_REPACK_Q6K=1` — measured neutral-to-negative by default). Q4_K_R4 measured +7% Pi / +5% M4 over plain multi-row. (2) **W6A8 SDOT Q6_K** (`dot_q6_k_row_q8_neon` + R4 variant): int8 activations, −32 folded into the int8 quants, one `sdot` per 16-element scale group — replaced the f32 widen/FMA path and was the single biggest CPU win (+21–25% Pi, +30–50% M4; Q6_K was arithmetic-bound, not layout-bound). Accuracy class = the accepted Q4_K W4A8 path (per-32-block activation scales). (3) **i8mm SMMLA prefill kernel** (`dot_q4_k_4rows_r4_x2_smmla`, ARMv8.6): two activation rows per pass through each R4 weight group — one `smmla` (32 MACs) replaces four `sdot`s; engaged in `matmul_nt_q4_k_r4` when `m ≥ 2 && i8mm` (M4/Grace have i8mm; Pi 5/A76 does not). **`smmla`'s 2×2 shape cannot help m=1 decode** (needs two distinct activation rows or half the lanes are waste) — decode stays on `sdot`. Prefill measured: M4 1.29×, Thor 1.68× (cumulative). Every kernel is bit-identity-gated against the single-row/scalar reference (`q4_k_4rows_matches_single_row`, `q4_k_r4_kernel_matches_single_row`, `q6_k_w6a8_neon_matches_scalar`, `q4_k_smmla_x2_matches_single_row`) and the engine-level `tests/cpu_repack.rs` proves bit-identical logits repack-on vs repack-off. Cumulative vs v0.5.0: **llama.cpp CPU decode gap 1.8–3.8× → ~1.2–1.35×** (see docs/BENCHMARKS.md). (4) **Q8_K-style precomputed activation block-sums** (`i8_block_sums`, parity round 2): the per-32-block Σx needed for Q4_K's `dmin·mn` term is computed ONCE per activation row at quantization time and passed into every W4A8 kernel (scalar + NEON single-row/4-row/R4/SMMLA×2) — the in-kernel `vaddlvq_s8` reductions ran per weight row/group. Bit-identical (same integer sums); measured +0.9% (llama-1B) / +1.2% (qwen-1.5B) M4 CPU decode, interleaved A/B. If you add a W4A8-family kernel, take `x_sums` as a parameter — don't re-reduce Σx in the inner loop. (5) **Decode spin/park threadpool** (`spinpool.rs` + `for_each_out_chunk` in matmul.rs): all decode GEMV parallel regions dispatch through a persistent worker pool instead of per-GEMV rayon fork/join (~230 barriers/token). Invariants: the op-handoff seqlock must bump the generation ODD **before** draining `active` (drain-first has a torn-slot race → SIGSEGV; stress-tested in `rapid_ops_with_constant_parking`); hot atomics are cache-line padded (`Pad<T>`, 128 B); workers AND publishers QoS-pin on macOS; the spin budget default is **4_000 iterations (~15 µs) — measured optimum on M4; long spins actively hurt** (serial-phase power/scheduler interference: 200k spins = 38.6 tok/s vs 63 rayon). Chunk geometry is identical to the rayon path → bit-identical results. `SAPIENT_SPINPOOL=0` (escape hatch/A/B), `_WORKERS`, `_SPINS`, `_DEBUG` (dispatch census). Thermal-governed decode falls back to rayon (`enabled()`), and `gemv_chunk`'s governed check compares within RAYON's domain — comparing against the pool's parallelism made `RAYON_NUM_THREADS=1` look governed (1-chunk serial collapse). Measured (M4, interleaved A/B): llama-1B parity (62.1 vs 62.9), qwen-1.5B +5.3% (45.3 vs 43.0); the Linux/ARM validation (Pi/Thor futex fork/join, where the 1.6× scaling gap was measured) is pending.

**SDOT Q8_0 (v0.3.x):** `dot_q8_0_block_sdot` uses `core::arch::asm!` inline assembly to emit the ARMv8.4A `sdot` instruction. Marked `#[target_feature(enable = "neon,dotprod")]` — the `dotprod` feature is required by the assembler even though the call site already gates on `is_aarch64_feature_detected!("dotprod")`. Net gain is ~3% because Q8_0 GEMV is memory-bandwidth-bound (not compute-bound) on M-series UMA. This is a W8A8 path (it quantizes the activation row to int8 too); activations are quantized **per 32-element block** (`quantize_row_to_i8_blocks`), matching the weight blocks. A single per-row activation scale (the old behavior) is set by outlier activation channels — common in LLMs — and collapses every normal-magnitude value to ~0, producing garbage; per-block scaling confines an outlier's damage to its own block (this is what llama.cpp does). Only Q8_0-*weight* models hit this path (online-quantized safetensors, or Q8_0 GGUFs); Q4_K GGUFs dequantize to F32 at load and never quantize activations. Tests: `sdot_q8_0_row_blockwise_survives_activation_outlier`, `matmul_nt_q8_0_matches_float`, `matmul_nt_q8_0_gguf_dimflip_matches_float`.

**Do not** add `-C target-cpu=native` to `.cargo/config.toml` for the `aarch64-apple-darwin` target — it causes `ring`'s compile-time const assertions to fail on CI runners.

### Thermal-aware sustained decode (Phase 8.4, `sapient-backends-cpu/src/thermal.rs`)
Passive Pis hit the 85 °C firmware trip under sustained decode and every core hard-throttles. The `ThermalGovernor` (hysteresis state machine, unit-tested against a fake sysfs root) samples the hottest `/sys/class/thermal/thermal_zone*/temp` and steps an **effective thread target** down one core at/above 80 °C (floor: half the cores — graceful, never collapse) and back up at/below 70 °C. The lever is `gemv_chunk()`: while governed it splits work into **exactly `effective_threads()` tasks with no chunk cap** — fewer tasks than pool threads leaves the surplus rayon workers nothing to steal (pools can't resize at runtime). **Pi-soak-validated trap:** the normal ×4 task multiplier and the 512-row chunk cap both silently defeat the governor (16 tasks / 297 lm_head tasks keep all 4 cores busy at any target) — the first on-device soak showed engagement with zero effect until both were bypassed in governed mode. Measured on the Pi 5: backoff costs only ~5% decode (8.70→8.23 tok/s, 0.5B) because decode is memory-latency-bound — shedding cores cuts package power nearly for free. Sample point: `thermal::tick()` at the top of the `matmul_nt` dispatcher, rate-limited to one sysfs read per 500 ms (all other calls are one atomic compare). Env: `SAPIENT_THERMAL=off`, `SAPIENT_THERMAL_HOT`/`_COOL` (°C), `SAPIENT_THERMAL_PATH` (fake sysfs for tests). Inert on machines without thermal zones (macOS/Windows/containers). See `docs/PI.md` for the user-facing story.

### Embedding lookup is row-wise — NEVER whole-table (Phase 8.3 finding)
`embed_tokens` (`forward/common.rs`) gathers rows via `gather_row_f32`, dequantizing/converting **only the requested rows**. The old code called `weight.to_f32_cow()` per forward call — a silent full-table dequant for quantized GGUF embeddings, ~0.8–1 GB of Q6_K work per decoded token on Llama-3.2-1B (tied 128k-vocab embed): the dominant decode cost on every quantized-embed model, hiding for months because qwen2.5-0.5b (the usual benchmark model) ships an unquantized embed and was unaffected. Fixing it: M4 CPU llama-1b 6.6→38.7 tok/s (5.9×), qwen-1.5b 11.5→33.5 (2.9×); Pi 5 llama-1b 1.3→8.3 (6.4×), qwen-1.5b 1.9→6.7, llama-3b 0.8→3.4. Row slices are bit-identical to full-dequant rows (per-row block layout; regression test `embed_row_gather_matches_full_dequant`). If you touch embedding paths, preserve row-wise access — a `to_f32_cow()`/`to_f32_vec()` on a whole quantized table in a per-token path is this bug reborn.

### Low-RAM GGUF quant override (Phase 8.2)
`SAPIENT_GGUF_QUANT=<tag>` (e.g. `Q4_K_S`) makes `select_best_gguf` pick the matching file (case-insensitive substring, best-scoring among matches) instead of the default Q4_K_M preference — for 4 GB boards where the `_S` variant's smaller file matters. Falls back to normal scoring with a warning when nothing matches. Pure selection core `select_gguf_with_override` is unit-tested without env vars.

### Q8_0 KV cache (in-place, zero allocation)
The KV cache is allocated as `DType::Q8_0` blocks (4× RAM reduction vs F32 for long contexts). Each decode step writes new K/V slices directly into the cache via `Tensor::as_bytes_mut()` — no per-step heap allocation. `as_bytes_mut()` is only valid on non-mmap tensors; the cache is always heap-allocated, so this invariant holds.

### Flash-Edge attention
`kernels/attention.rs` implements an online-softmax tiled attention algorithm that never materialises the full seq_q × seq_k score matrix. Working memory is O(head_dim). Uses NEON `vfmaq_f32` on `aarch64`. This replaces the previous naive `scaled_dot_product_attention` for the live-chat path.

### Tensor API additions (v0.2.9)
- `Tensor::from_f32_vec(Vec<f32>, shape)` — wraps a `Vec<f32>` as a tensor with zero copy (takes ownership).
- `Tensor::as_bytes_mut()` — mutable byte slice into the underlying buffer for in-place quantized writes.
- `CpuBuffer::from_f32_vec(Vec<f32>)` — low-level counterpart; wraps without copying.

### Online quantization at load time
F16/BF16 safetensors weights are auto-quantized to Q8_0 during loading (near-lossless, ~1.06 bytes/weight). This eliminates the F16→F32 expansion that previously dominated memory bandwidth for safetensors models.

### Speculative decoding
`SpeculativePipeline` wraps a draft and a target `Pipeline`. The draft proposes K tokens; the target verifies them in one cache-appending forward pass; accepted tokens are kept (rejection sampling corrects the distribution). Exposed via `sapient chat --speculative [--draft-model <alias>]` and `sapient serve --speculative [--draft-model <alias>]`.

**Engine reuse (serve-ready).** Like `Pipeline`, it reuses the loaded target+draft engines across requests: the streaming/non-streaming paths clone `self.target.engine_arc()` / `self.draft.engine_arc()` (`Arc<Mutex<ForwardEngine>>`) and **lock** them inside `spawn_blocking` — they do NOT rebuild engines via `from_weight_paths_with_backend` per request (that re-quantized the whole model and dominated TTFT). Has full `*_with_config` methods + `tokenizer()`/`arch()`/`is_mmap()`/`config()` accessors + `new_with_opts`/`with_auto_draft_with_opts` (backend/mmap). In `serve`, a resident model is `ServedModel::{Plain(Pipeline), Speculative(SpeculativePipeline)}` (in `server.rs`) and the LRU cache treats both uniformly.

**Cache-aware verification (critical invariant).** Target verification MUST use `forward_all_logits_cached` (NOT `forward_all_logits`). `forward_all_logits` calls `forward_hidden(.., use_cache=false)` which **resets the KV cache and starts at position 0** — verifying drafts with no prompt context, producing token-salad. This was a real bug (all speculative output was garbage). `forward_all_logits_cached` does `forward_hidden(.., true)`: it appends the draft tokens to the target KV (positions continue from the prompt) and returns per-position logits. The `spec_round` loop then rolls back rejected speculative tokens with `truncate_cache(n)`, maintaining the invariant "both target and draft KV caches hold exactly the committed tokens" and carrying each model's next-token logits across rounds. (MLX `truncate_cache` resets to 0 → no incremental rollback, so speculative uses the Llama/Phi CPU engines.)

**Vocab guard + family-aware auto-draft.** Draft and target must share a vocabulary (the draft proposes token IDs the target scores with its own logits). `new_with_opts` errors on a vocab mismatch instead of emitting garbage; `with_auto_draft` (`select_auto_draft`) picks a draft from the **same family** as the target (Qwen→`qwen2.5-0.5b`, SmolLM2→`smollm2-135m`).

### OpenAI-compatible HTTP server (`sapient serve`)
`server.rs` in `sapient-cli` exposes a chat-completion server with the following endpoints:
- `GET /v1/models` — list loaded model(s)
- `POST /v1/chat/completions` — OpenAI-compatible chat
- `POST /v1/completions` — raw completion
- `GET /v1/health` — liveness check

No model is loaded at startup; the first API request triggers download + load (Ollama-style lazy loading).

**Multi-model LRU cache (Phase 1, beats Ollama's single-resident model).** `ModelCache` in `server.rs` keeps the N most-recently-used models resident (`--max-models`, default 3) bounded by a RAM byte budget (`--cache-gb`, default ~70% of system RAM); switching back to a recent model is a cache hit (no download / re-quant / engine rebuild) instead of Ollama's cold reload. Design notes: each entry is `Arc<CachedModel>` so a streaming request keeps its model alive even if it's evicted mid-stream; `get_or_load` never holds the cache lock during the (slow) load or during inference, so cache hits and *other* models' requests run concurrently (only same-model inference serializes, on the engine's internal mutex); a `load_lock` serializes loads to prevent double-loading the same model on concurrent first-requests. Eviction is by both count (`max_models`) and bytes (LRU front evicted until both fit); size is estimated from on-disk weight size (`hub::cached_model_size`). `/v1/health` and `/v1/models` report `resident_models` + `active_model` (MRU). Measured: ~5× faster model switch-back on cache hit (cold reload of a large model is far higher — see deep-research notes).

**Admission control + prefix caching (Phases 3-4).** `inference_sem` (tokio `Semaphore`, `--max-concurrency`) bounds concurrent inferences. **Prefix/prompt KV caching**: `Pipeline::enable_prefix_cache()` (serve enables it) makes generation reuse the KV cache for the longest *token* prefix shared with the previous call (`ForwardEngine::truncate_cache(n)` + `Pipeline.last_prompt` + `common_prefix_len`), instead of `reset_cache()` + full re-prefill — multi-turn chat / shared system prompts skip re-prefilling history. Correct because only matching token IDs are reused (KV at `[0..P]` is identical → byte-identical greedy output) and same-model calls are serialized on the engine lock; off by default so CLI chat is unchanged. MLX `truncate_cache` falls back to reset (no reuse, still correct). **Deferred** (need engine work): speculative-in-serve (SpeculativePipeline rebuilds engines per request — needs engine-reuse refactor first), continuous batching + PagedAttention (engine is single-sequence: `forward_logits(&[u32])`, KV `[1,n_kv,max_seq,hd]` — needs a batched-forward + block-pool rewrite).

### Hybrid Metal+CPU inference (v0.3.x)
Both `LlamaForward` and `PhiForward` support layer-split hybrid execution. `compute_backend_split()` / `compute_phi_backend_split()` run at model load and decide: full Metal, hybrid split, or CPU-only based on `(model_bytes × 1.5 ≤ RAM − 2 GB)`. The `forward_layer` is structured in **three borrow-safe phases** to satisfy the Rust borrow checker:
1. Pre-cache phase — borrows `&self.backend` (or `&self.cpu_fallback`) to compute norm, QKV, RoPE.
2. Cache phase — borrows `&mut self.cache[layer_idx]` (backend ref dropped).
3. Post-cache phase — re-borrows backend for attention + FFN.
Helper functions (`linear_with_bias_bk`, `mlp_phi2_bk`, `mlp_phi3_bk`) take explicit `bk: &LlmBackendDispatch` and `weights: &HashMap` so individual fields are borrowed rather than all of `self`.

### Metal SDPA head_dim compatibility
`mlx_sdpa_supported_head_dim(head_dim)` returns true only for {32, 64, 96, 128, 256}. MLX pre-compiles Metal SDPA shaders for this fixed set; any other value (e.g. Phi-2's 80) panics at runtime. `LlmBackendDispatch::from_kind_with_head_dim()` checks this at init: Auto silently falls back to CPU; explicit `--backend metal` returns a user-readable error.

### One-shot chat (`sapient chat --prompt`)
`sapient chat <model> --prompt "<text>"` (`-p`) runs a **single chat turn** and exits instead of entering the REPL. Critically it goes through the **chat path** (`Pipeline::chat` → `format_chat_prompt` applies the model's chat template + end-of-turn EOS), so an instruct model returns a clean, bounded answer. This is different from `sapient run --prompt`, which calls `Pipeline::generate` on the **raw** prompt (no template, no turn-stop) → the model continues/ rambles and can loop (e.g. a 0.5B model degenerating into repeated text). The one-shot path prints **only the reply to stdout** (status/spinners stay on stderr), so it's scriptable — e.g. `reply=$(sapient chat qwen2.5-0.5b-q4 -p "Tell me a joke"); sapient speak kokoro-82m "$reply"` (this is exactly what `scripts/sts_test.py` does for mic-free text→voice testing). `--prompt` takes precedence over `--speculative` (one-shot uses the standard engine).

### Chat input line editor (bracketed paste)
The interactive `sapient chat` REPL reads input via `rustyline::DefaultEditor` (`read_chat_line` in `sapient-cli/src/main.rs`), **not** `stdin().read_line()`. Plain `read_line` returns the instant it sees a newline, so any pasted text containing or ending with `\n` was submitted before the user pressed Enter (the v0.3.x paste bug). rustyline enables bracketed-paste mode by default, so a paste is inserted into the edit buffer as literal text (newlines included) and only a real Enter submits. `read_chat_line` returns `Ok(None)` on EOF / Ctrl-C / Ctrl-D (caller breaks) and feeds non-empty lines to history. The prompt is passed to `readline()` as `ui::user_prompt_str()` (styled string, ANSI handled by rustyline's cursor math). Both `chat_command` and `chat_speculative_command` share this path; piped/non-TTY input still works via rustyline's line-by-line fallback.

### Live Markdown rendering of replies (`markdown.rs`)
`sapient-cli/src/markdown.rs` (`StreamRenderer`) renders the assistant's streamed Markdown **as it streams** — prose/headings/lists/tables/inline styling via `termimad`, fenced code blocks via `syntect` 24-bit syntax highlighting (dim `│` gutter per line). Both chat loops route every token through `renderer.push(&token)` instead of `print!`.

**Commit-and-preview streaming.** Repainting the whole reply per token thrashes the screen and breaks once output scrolls past the viewport top (cursor can't move back up). Instead the renderer splits the reply into Markdown *blocks* (`complete_prefix_len`): **completed** blocks (separated by a blank line, or closed by a code fence) are rendered once and printed permanently; the **trailing incomplete** block is repainted in place each update via `\x1b[{n}A\r\x1b[0J` (move up over the preview, clear, reprint). `cursor_down_moves` computes the preview's row count *wrap-aware* (uses `console::measure_text_width`, which strips ANSI) so the cursor-up count is exact even when lines soft-wrap. A viewport guard commits the preview early if one in-progress block would exceed the screen height (rare, e.g. a huge unclosed code block). Repaints are throttled to ~30 fps unless a token contains `\n`.

**Raw / non-TTY fallback.** Rich rendering is disabled — and tokens stream as plain text — when stdout is not a terminal, when `NO_COLOR` is set, or with the `sapient chat --raw` flag. This keeps `sapient chat | tee log.txt` clean. Unit tests in `markdown.rs` cover block-boundary detection, wrap-aware row counting, and that `render()` emits ANSI styling + the code gutter.

### Stop-sequence handling
The streaming generator (`generate_stream`) buffers decoded text and withholds up to `max(stop_len)` bytes from the tail before emitting, preventing stop markers from leaking. Both EOS-by-token-id (multi-EOS, all candidates collected from the tokenizer vocab) and EOS-by-string are checked every step.

## Adding a new model architecture

1. Check if it's Llama-compatible (RMSNorm, SwiGLU, standard RoPE) — if so, just add a registry entry and it runs through `LlamaForward`.
2. If it needs a distinct forward pass: add `crates/sapient-models/src/forward/<arch>.rs`, add a variant to `ForwardEngine`, update `forward/mod.rs` dispatch, then add registry entries.
3. For GGUF: check that `map_gguf_tensor_name` in `gguf_weights.rs` covers the tensor naming; add `.bias` suffixes if the arch has projection biases.

## Version and release

- Version is set once in workspace `Cargo.toml` (`[workspace.package] version`); all crate `Cargo.toml` files inherit it via `.workspace = true`. Internal workspace deps still carry a matching `version = "x.y.z"` (kept in sync; required for path+version deps even though we don't publish).
- Release is triggered by pushing a `vX.Y.Z` tag; the workflow in `.github/workflows/release.yml` builds all platform binaries including a `-metal` variant for Apple Silicon (MLX) and **`-gpu` variants** for x86_64 Linux (Vulkan), x86_64 Windows (DX12), **and x86_64 macOS (wgpu→Metal — the GPU path for Intel Macs with AMD/Intel GPUs, where the Apple-Silicon-only MLX `-metal` build can't run; cross-compiled to x86_64 on the `macos-14` ARM runner)**, all built with `--features wgpu`. **aarch64-linux builds natively on a GitHub-hosted ARM runner (`ubuntu-22.04-arm`, GA since 2025)** with default features so cpal/ALSA compile and `converse` ships on the Pi (the older `cargo zigbuild` cross had to drop `audio-io`; the GA ARM runners are stable, unlike the earlier preview pool).
- **Build-variant selection (install + update).** `install.sh`/`install.ps1` auto-detect a GPU on x86_64 (Linux: `/dev/dri/renderD*` or `vulkaninfo`; Windows: `Win32_VideoController`; **Intel macOS: always — every Mac has a Metal GPU**) and download the `-gpu` build when present, else CPU; override with `SAPIENT_VARIANT=cpu|gpu`. `sapient update` (`update.rs`) has `Variant::{Cpu,Metal,Gpu}`; `resolve_variant` offers the accelerated build interactively only when the machine is `metal_capable()` (Apple Silicon) or `gpu_capable()` (x86_64 Linux/Windows with a GPU, **or any Intel Mac**), or takes `--metal`/`--gpu`/`--cpu`. `current_variant()` reads the compiled feature (`mlx`→Metal, `wgpu`→Gpu, else Cpu). **Intel Mac = `-gpu` (wgpu/Metal), NOT `-metal` (MLX is Apple-Silicon-only).**
- **SAPIENT is NOT published to crates.io.** Distribution is prebuilt GitHub release binaries (+ install script / Homebrew). The previously-published crates (0.1.11–0.3.1) have been yanked; `scripts/yank-all.sh` (idempotent, `--undo` to reverse) manages crates.io yanks. Do not re-introduce a publish step.
